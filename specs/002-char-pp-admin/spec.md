## 背景

`char_pp` 是乒乓球AI智能教练系统的后端服务（Python/FastAPI），已实现 11 个功能规范，提供视频分析、教练知识库、动作诊断等核心能力。本后台管理系统（pp_admin，Vue 2 + Element UI）作为 char_pp 的运营管理界面，直接调用 char_pp 暴露的 REST API（`/api/v1/...`），覆盖 char_pp 已有功能的运营监控与数据管理需求。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 任务总览与状态监控 (优先级: P1)

运营人员需要实时了解系统中所有视频分析任务（专家视频 + 运动员视频）的处理状态，以便发现异常任务、追踪进度、排查故障。

**优先级原因**: 任务监控是后台管理的核心入口，其他所有功能（诊断、知识库、标准）都通过任务体系流转，任务列表可独立交付运营价值。

**独立测试**: 可通过访问任务列表页面、查看状态过滤、点击单条任务详情完整验证，无需其他模块配合。

**验收场景**:

1. **给定** 系统中存在多条分析任务，**当** 运营人员访问任务列表页，**那么** 列表展示任务ID、视频路径、任务类型（专家/运动员）、状态（pending/processing/completed/failed）、创建时间、处理耗时，并支持按状态、类型分页过滤。
2. **给定** 运营人员点击某条任务，**当** 打开任务详情，**那么** 展示完整的视频元数据（时长、帧率、分辨率）、关联教练信息、以及处理结果摘要（专家视频显示提取的技术点数量；运动员视频显示偏差报告概览）。
3. **给定** 任务状态为 failed，**当** 查看详情，**那么** 展示错误原因，并提供操作提示。

---

### 用户故事 2 - 教练管理 (优先级: P1)

运营人员需要维护教练档案，包括新增教练、修改信息、停用不再使用的教练，并能为已有分析任务关联正确的教练。

**优先级原因**: 教练是知识库和标准构建的核心数据源，教练数据准确性直接影响 AI 系统输出质量。

**独立测试**: 可通过教练列表的增删改查操作独立验证，无需依赖其他模块。

**验收场景**:

1. **给定** 运营人员在教练管理页点击「新增教练」，**当** 填写教练名称和简介并提交，**那么** 系统成功创建教练记录，列表中出现新条目。
2. **给定** 教练名称与现有记录重复，**当** 提交新增，**那么** 系统提示名称重复，不创建。
3. **给定** 运营人员点击「停用」某教练，**当** 确认操作，**那么** 该教练变为停用状态，列表可通过「含停用」开关查看历史记录，但停用教练不可被新任务引用。
4. **给定** 运营人员在任务详情页为某任务关联教练，**当** 选择有效的活跃教练并提交，**那么** 任务与教练关联成功，任务详情中展示教练信息。

---

### 用户故事 3 - 视频分类管理 (优先级: P2)

运营人员需要查看 COS 中所有视频的分类结果，对分类不准确的视频进行手动纠正，并批量触发专家视频的知识提取任务。

**优先级原因**: 视频分类准确性是知识库质量的前提，人工纠错是保证数据质量的关键运营操作。

**独立测试**: 可通过分类列表查看、手动覆盖某条记录、触发刷新分类操作独立验证。

**验收场景**:

1. **给定** 运营人员访问视频分类页，**当** 页面加载，**那么** 展示所有视频的分类信息（教练名、技术类别、子类别、技术细节、动作类型、视频类型、置信度、是否手动覆盖），支持多条件过滤。
2. **给定** 某视频分类有误，**当** 运营人员在列表中手动修改该视频的分类字段并提交，**那么** 该记录标记为「手动覆盖」，后续自动重新扫描不会覆盖此记录。
3. **给定** 运营人员点击「重新扫描分类」，**当** 操作执行，**那么** 系统触发全量重新分类（跳过已手动覆盖的记录），并展示扫描结果（重新分类数量/跳过数量/总扫描数量）。
4. **给定** 运营人员在分类列表按条件筛选后点击「批量提交知识提取」，**当** 选择音频分析选项并确认，**那么** 系统为符合条件的视频批量创建知识提取任务，弹窗展示已提交的 task_id 列表（每个条目可点击跳转到任务详情页）。

---

### 用户故事 4 - 技术标准管理 (优先级: P2)

运营人员需要查看各技术类别的当前有效标准（ideal/min/max 参数），了解标准质量，并按需触发标准重建。

**优先级原因**: 技术标准是动作诊断的比对基准，运营人员需要能够验证标准数据的完整性和覆盖范围。

**独立测试**: 可通过标准列表查看各维度参数、过滤缺失类别、触发重建操作独立验证。

**验收场景**:

1. **给定** 运营人员访问技术标准页，**当** 页面加载，**那么** 展示所有活跃技术类别的标准摘要（技术类别、版本、质量类型、覆盖教练数、维度数、构建时间），并标注尚无标准数据的技术类别。
2. **给定** 运营人员点击某技术类别，**当** 查看标准详情，**那么** 展示各维度的 ideal/min/max 值、单位和样本数量。
3. **给定** 运营人员需要更新某技术类别的标准，**当** 点击「重建标准」并指定（或不指定）技术类别，**那么** 系统触发单类别或批量重建，展示构建结果（成功/跳过/失败 原因）。

---

### 用户故事 5 - 动作诊断管理 (优先级: P2)

运营人员需要向系统提交视频进行动作诊断（用于测试验证），并查看诊断报告详情，验证诊断结果的准确性。

**优先级原因**: 诊断是 char_pp 最新实现的核心功能（spec 011），后台需要提供便捷的测试入口，替代直接调用 API。

**独立测试**: 可通过填写诊断表单、提交已有视频路径、查看报告详情独立验证。

**验收场景**:

1. **给定** 运营人员在诊断管理页填写视频路径和技术类别，**当** 提交诊断请求，**那么** 系统展示全屏加载遮罩和进度提示文案，同步等待 char_pp 返回结果（最长 60 秒），完成后展示包含综合评分、各维度测量值/标准对比、优点列表、改进建议的完整报告。
2. **给定** 诊断请求对应的技术类别无有效标准，**当** 提交，**那么** 系统展示明确的错误提示「该技术类别暂无有效标准数据」。
3. **给定** 诊断完成后，**当** 运营人员查看报告详情，**那么** 各维度卡片展示测量值与 ideal/min/max 对比、偏差等级标签（正常/轻度偏差/明显偏差）及改进建议文字。
4. **给定** 诊断请求超过 60 秒未返回，**当** 超时触发，**那么** 系统关闭加载遮罩，展示「请求超时，点击重试」按钮，保留用户已填写的视频路径和技术类别输入，不清空表单。

---

### 用户故事 6 - 知识库与教学提示查看 (优先级: P3)

运营人员需要查看已构建的技术知识库内容（各版本的技术要点）和教学提示（Teaching Tips），了解 AI 系统积累的知识资产。

**优先级原因**: 知识库内容查看是只读监控功能，对运营人员了解系统能力有帮助，但不阻塞核心运营流程。

**独立测试**: 可通过知识库列表和教学提示列表的查询操作独立验证。

**验收场景**:

1. **给定** 运营人员访问知识库页，**当** 页面加载，**那么** 展示按技术类别分组的知识库版本列表，可查看每个版本包含的技术要点数量、来源教练数量。
2. **给定** 运营人员访问教学提示页，**当** 按技术类别过滤，**那么** 展示该类别下所有教学提示的内容、来源视频片段和置信度。

### 边界情况

- char_pp 后端服务不可用时，展示「服务暂不可用」横幅，同时显示本次会话内上次成功加载的数据（只读，禁用所有写操作按钮）；恢复连接后自动解除只读限制。
- 诊断请求超过 60 秒未返回时，前端如何处理超时？
- 批量提交知识提取任务后，以弹窗形式展示已提交的 task_id 列表，每条可点击跳转到任务详情页，不轮询进度。
- 视频路径包含中文或特殊字符时，分类 PATCH 接口的 URL 编码处理。
- 运营人员尝试停用已关联活跃任务的教练时，系统应如何提示？（当前接口允许停用，仅阻止新任务关联）

## 需求 *(必填)*

### 功能需求

| 编号 | 需求 |
|------|------|
| **FR-001** | 后台管理系统必须通过 char_pp 的 REST API（`/api/v1/...`）获取所有业务数据，不直接访问数据库。 |
| **FR-002** | 系统必须提供任务列表页，展示任务状态、类型、创建时间，并支持按状态/类型分页过滤，对应接口：`GET /api/v1/tasks`（需在 char_pp 中新增此列表接口；在接口就绪前，US1 作为占位页先交付）。 |
| **FR-003** | 系统必须提供教练 CRUD 管理：新增（`POST /api/v1/coaches`）、列表（`GET /api/v1/coaches`）、编辑（`PATCH /api/v1/coaches/{id}`）、停用（`DELETE /api/v1/coaches/{id}`）。 |
| **FR-004** | 系统必须支持为任务关联教练，对应接口：`PATCH /api/v1/tasks/{task_id}/coach`。 |
| **FR-005** | 系统必须提供视频分类列表查看，支持多条件过滤，对应接口：`GET /api/v1/videos/classifications`。 |
| **FR-006** | 系统必须支持手动覆盖视频分类，对应接口：`PATCH /api/v1/videos/classifications/{cos_object_key}`。 |
| **FR-007** | 系统必须支持触发视频重新分类扫描，对应接口：`POST /api/v1/videos/classifications/refresh`。 |
| **FR-008** | 系统必须支持批量提交知识提取任务，对应接口：`POST /api/v1/videos/classifications/batch-submit`。 |
| **FR-009** | 系统必须提供技术标准列表（`GET /api/v1/standards`）、详情（`GET /api/v1/standards/{tech_category}`）和重建触发（`POST /api/v1/standards/build`）功能。 |
| **FR-010** | 系统必须提供动作诊断提交表单（`POST /api/v1/diagnosis`）和诊断报告详情展示，包含综合评分、各维度对比卡片和改进建议。 |
| **FR-011** | 所有 API 调用错误（4xx/5xx）必须以中文友好提示展示给用户，不暴露原始错误码细节。 |
| **FR-012** | 系统通过现有的 RBAC 权限机制控制菜单访问，super_admin 和 admin 均可访问所有模块，两者权限无差异；后台管理菜单对两个角色完全开放。 |

### 关键实体

- **AnalysisTask（分析任务）**: 代表一次视频分析请求，包含任务类型（专家/运动员）、状态、视频元数据、关联教练，是系统的核心流转单元。
- **Coach（教练）**: 代表系统中的教练档案，是知识库和标准构建的数据来源，具有活跃/停用状态。
- **VideoClassification（视频分类）**: 代表 COS 中某个视频的技术分类结果，可被手动覆盖以防止自动刷新覆盖。
- **TechStandard（技术标准）**: 代表某技术类别的当前有效比对基准，包含各维度的 ideal/min/max 参数。
- **DiagnosisReport（诊断报告）**: 代表一次动作诊断的完整结果，包含综合评分、维度分析和改进建议。

- Q: char_pp 服务不可用时，后台页面的降级策略？ → A: 展示「服务暂不可用」横幅，同时显示本次会话内上次成功加载的数据（只读，不允许写操作）。
- Q: 批量提交知识提取任务后，运营人员如何跟进这批任务的处理进度？ → A: 弹窗展示已提交的 task_id 列表，每个条目可点击跳转到任务详情页。

## 假设

- char_pp 服务通过环境变量 `VUE_APP_CHAR_PP_API` 配置 API 基础地址，开发环境代理到本地运行的 char_pp 服务。
- char_pp 当前无 `GET /api/v1/tasks`（任务列表）接口；已确定在 char_pp 中新增该接口，后台 US1 当前作为占位页先交付，char_pp 接口就绪后对接完整功能。
- 诊断功能的同步等待最长 60 秒，前端展示全屏加载遮罩 + 进度提示文案；超时后保留表单输入，展示「请求超时，点击重试」按钮。
- 视频分类中的 `cos_object_key` 可能包含路径分隔符（如 `coaches/zhang/video.mp4`），PATCH 请求需进行 URL 编码。
- 本后台仅作为 char_pp 的运营管理界面，不独立存储业务数据，所有状态以 char_pp 数据库为准。
- char_pp 目前支持两种技术动作类型：`forehand_topspin`（正手拉球）和 `backhand_push`（反手推挡），后续随 char_pp 扩展而自动覆盖。
- 现有 RBAC 权限体系（super_admin/admin）复用，不新增角色；两个角色权限相同，均可访问所有后台模块。
- 后台管理不实现用户账户管理功能（char_pp 本身是匿名诊断模式，无用户体系）。
